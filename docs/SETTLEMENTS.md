# Settlements & Payouts

Settlement tracks how much each organizer is owed from ticket sales and lets them request payouts. It is backed by a double-entry-style ledger (`SettlementLedger`) and exposed through organizer-only dashboard routes.

> **✅ Status: active (verified 2026-06-27).** This subsystem is fully wired:
> - `routes/settlement-routes.php` **is loaded** — `bootstrap/app.php` registers it in the `withRouting(then: ...)` callback under the `api` middleware and `api` prefix, so the endpoints below are reachable at `/api/organizer/settlement/*`.
> - **The ledger is written on every paid payment.** `PaymentController::callback()` calls `SettlementService::recordSuccessfulPayment()` (gross credit + 10% platform-fee debit) inside a DB transaction when a gateway verifies a payment as paid.
> - **Payouts move money deliberately.** `request-payout` validates the requested amount against the available balance and creates a `pending` `Payout`; an admin approval calls `Payout::markCompleted()` (idempotent), which records the payout debit. `Payout::reject()` closes the request. Both completion and rejection also write an audit-log entry (see [`ADMIN_PANEL.md`](ADMIN_PANEL.md)).
> - Coverage: `tests/Feature/SettlementLedgerTest.php` (fee math, paid-callback ledger entries, pending-payout balance, idempotent payout completion, monthly statement).

## Settlement Ledger

`App\Models\SettlementLedger` records every balance-changing event for an organizer. Each row stores `amount`, `type` (`credit`/`debit`), `reference_type` + `reference_id` (e.g. `payment`, `payout`), a `description`, and the running `balance_before` / `balance_after`.

Helper recorders:

| Method | Effect |
|---|---|
| `recordPayment(organizerId, paymentId, amount)` | Credit — funds received from a paid registration. |
| `recordPlatformFee(organizerId, paymentId, fee)` | Debit — platform fee (10%). |
| `recordPayout(organizerId, payoutId, amount)` | Debit — funds paid out to the organizer. |

`getBalance(organizerId)` derives the current balance from the ledger; each recorder snapshots `balance_before`/`balance_after` so the history is auditable.

## Dashboard Endpoints

Defined in `routes/settlement-routes.php` under the `organizer` prefix and `auth:sanctum`, served by `SettlementDashboardController` via `SettlementService`:

| Endpoint | Method | Purpose |
|---|---|---|
| `/organizer/settlement/dashboard` | GET | Balance summary, recent payouts (paginated), current monthly statement, and `can_request_payout`. |
| `/organizer/settlement/request-payout` | POST | Request a withdrawal. |
| `/organizer/settlement/statements` | GET | Settlement statements. |
| `/organizer/settlement/ledger` | GET | Raw ledger entries. |

Requests fail with `403` if the authenticated user is not linked to an organizer.

### Balance & Payout Rules

- Balance is computed by `SettlementService::calculateOrganizerBalance($organizer)` and exposes an `available` figure.
- The minimum payout is **100,000 IRR**; `can_request_payout` is true only when `available >= 100000`.
- `request-payout` validates `amount` (integer, `min:100000`), `bank_account` (required), and optional `notes`, and rejects amounts above the available balance with `422`.
- Monthly statements are generated by `SettlementService::generateMonthlyStatement($organizer, 'Y-m')`.

## Relationship to Payments

The credit/fee ledger entries originate from verified gateway payments — see [`PAYMENTS.md`](PAYMENTS.md). All amounts are integer Rials (IRR).
