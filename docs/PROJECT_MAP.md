# Project Map (Ground Truth)

A reality-scan of the repository as it actually exists — counts, components, and notable gaps between the existing docs and the code. Generated by inspecting the source tree directly, not from the roadmap. Use this as the orientation doc when onboarding; deep-dives live in the per-feature docs linked from [`INDEX.md`](INDEX.md).

## At a Glance

| Area | What's there |
|---|---|
| Backend | Laravel API — **20 controllers**, **30 models**, **19 Filament** classes, **41 test files** |
| Frontend | **Vite + Vue 3 + Tailwind v4**, packaged for Android via Capacitor. Not a typical multi-component SPA (see below). |
| Workers | Python package `rokhdad_workers` — ingestion, normalization, dedup, enrichment, images, Graphiti. **15 test files**. |
| Routes | `api.php` (128 lines), `console.php` (556), `settlement-routes.php`, `web.php` |
| Migrations | **27** migration files |
| Data stores | MariaDB (relational), MongoDB (raw payloads/snapshots), Redis (queue), Neo4j (Graphiti, optional) |
| Docs | 25 markdown files in `docs/` |

## Test Coverage

Backend tests track features closely (41 files): auth/OTP, RBAC, public events/detail/lookups/profiles, registration, payment gateway abstraction + flow, tickets + validation, comments/ratings, organizer dashboard, campaign manager, webhook subscription + delivery, attendee transfer, personalized homepage, Filament admin/resources, event-source health, field overrides, notifications/SEO, health endpoint.

This means **behavior is well-specified by tests** — they are the most reliable description of intent for most features.

## Notable Findings (docs ↔ code reality)

1. **Frontend is not a component-heavy SPA.** `frontend/src` contains a single `App.vue` plus plain-JS feature modules — `events/homepage.js`, `events/registration-form.js`, `organizer/dashboard.js`, `lookups/directory.js`, `api/client.js` — each with a co-located `*.test.js` run via `node --test`. Stack is Vite + Vue 3 + Tailwind v4 (`@tailwindcss/vite`), with a PWA asset step. Treat [`FRONTEND.md`](FRONTEND.md) as scaffold notes, not a full component map.

2. **Schedulers run every minute, not hourly.** `routes/console.php` registers `evand:import --pages=2` and `eseminar:import --pages=2` on `everyMinute()`. Some commit messages say "hourly sync" — the code says per-minute. `bilitmaster` has a worker-side source connector but **no scheduled artisan importer**.

3. **More console commands than the two class-based ones.** Besides `app:setup-admin-user` and `evand:historical-sync` (class commands), `console.php` defines inline `evand:import`, `evand:import-organizers`, and `eseminar:import`. See [`ADMIN_PANEL.md`](ADMIN_PANEL.md).

4. **Two campaign data designs coexist** (single `campaigns` table vs. `campaigns`+`campaign_messages`+`campaign_analytics`). See [`CAMPAIGNS.md`](CAMPAIGNS.md).

5. **Domain logic isn't only under `Services/`.** Only 3 classes sit under a `*Services*` path; substantial logic also lives in `app/Webhooks/`, notification provider classes, and gateway classes. Don't assume `app/Services` is the whole business layer.

6. **The settlement subsystem is dormant (verified by inspection).** `routes/settlement-routes.php` is never loaded by `bootstrap/app.php` (only `web.php` + `api.php` are), so its endpoints are unreachable; the `SettlementLedger` is never written to (no caller of `recordPayment`/`recordPlatformFee`/`recordPayout`, and `PaymentController` doesn't touch it); and there are no settlement tests. The code is scaffolding, not a live feature. See [`SETTLEMENTS.md`](SETTLEMENTS.md).

7. **Routing facts worth knowing.** API prefix is `api` with no version segment baked into `withRouting` (versioning is done inside `api.php` via the `v1` group). A framework health route also exists at `/up`. SEO routes (`/sitemap.xml`, `/robots.txt`) and a `welcome` view live in `web.php`.
